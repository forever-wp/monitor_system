事件仲裁器与事件执行层

这页用前端 HTML 的方式说明：事件发现器只发现事实，事件仲裁器按法典生成统一执行计划，事件执行层独立完成软端 nodemanager 与硬件 safety 两类执行。

发现不执行

事件发现器只发布事实 active / inactive，不再认定最终错误，也不直接触发恢复或执行。

执行永远独立

事件仲裁器只输出执行计划；事件执行层负责真实执行、执行反馈、锁存、幂等和软硬通道隔离。

1. 为什么需要独立仲裁层

事件发现器如果直接携带动作，多个事件一起出现时就会产生重复命令、软硬执行冲突、恢复顺序不清晰等问题。因此所有执行都必须由事件仲裁器统一计算执行计划，再交给事件执行层独立完成。

逐步出现

A 先出现，B 后出现。系统按当前事件事实集匹配法典，满足 A+B 后升级为组合执行计划。

同批出现

A 和 B 同一轮出现。高危组合应直接命中法典，而不是先发一堆分散执行命令。

交叉覆盖

A+B 和 B+C 同时命中时，要用稳定优先级选赢家，保证执行计划可复现。

一句话：事件发现器回答“现在发生了什么”，事件仲裁器回答“法典要求做什么”，事件执行层回答“实际执行结果如何”。

5. 多个错误逐步出现

逐步出现时，不要等组合才处理。先按单项错误处理，满足组合后再升级。

6. 多个错误同时出现

同一轮检测里 A 和 B 同时出现时，高危组合可以直接激活，避免先发多个单项动作再升级。

可视化流程图
输入 / 数据源判断 / 策略状态 / 输出故障 / 风险执行 / 安全动作
建议：高危组合 `enter_hold_s` 取 0 到 0.05 秒；普通组合取 0.1 到 0.3 秒；容易抖动的组合加长 `clear_hold_s`。

7. 组合策略交叉时谁优先

如果多个组合策略共享同一个基础错误项，必须用稳定规则选赢家。

可视化流程图
输入 / 数据源判断 / 策略状态 / 输出故障 / 风险执行 / 安全动作
顺序 判断项 说明
1priority显式优先级，高的优先。
2safety_systemEMERGENCY_STOP > SOFT_STOP > SLOW_DOWN > NONE。
3levelCRITICAL > ERROR > WARNING > NORMAL。
4matched_count覆盖基础错误更多的优先。
5已激活规则当前已激活的优先，避免边缘跳变。
6rule.name字典序兜底，保证结果稳定可复现。

组合恢复必须有状态机。否则在障碍边缘、topic 抖动、反馈恢复瞬间会反复跳变。

9. 推荐配置样例

配置既要能表达组合关系，也要能控制触发和恢复节奏。

event_codex_rules:
  - rule_id: "collision_and_navigation_unhealthy"
    name: "collision_and_navigation_unhealthy"
    priority: 90
    when_all:
      - module: "collision_detection"
        fault_type: "collision"
      - module: "navigation"
        fault_type: "topic"
    min_match_count: 2
    enter_hold_s: 0.1
    clear_hold_s: 0.5
    min_hold_s: 1.0
    execution_plan:
      targets: ["safety", "nodemanager"]
      safety:
        action: "SOFT_STOP"
      nodemanager:
        modules: ["navigation"]
    report_policy:
      severity: "CRITICAL"
      reason: "Collision risk and navigation degraded"
字段 作用
when_all所有条件都要命中。
when_any至少一个条件命中。
min_match_count至少命中多少个不同基础错误。
enter_hold_s进入组合前的确认时间。
clear_hold_s解除组合前的确认时间。
min_hold_s组合激活后的最小保持时间。
execution_plan组合命中后的最终审判结果，交由事件执行层执行。

10. 实现计划细化

组合策略仲裁已收敛为 EventCodexArbiter，不再通过 FaultDetector 附加虚拟组合故障。

第一步

由 EventCodexArbiter 统一处理 event_codex_rules，兼容旧 combined_fault_rules。

第二步

支持条件匹配：module、fault_type、fault_name、prefix、contains、action。

第三步

加入状态机、组合恢复、动作抑制和交叉策略优先级。
