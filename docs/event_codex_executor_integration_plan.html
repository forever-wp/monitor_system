monitor_system / 事件法典架构计划

事件法典仲裁与事件执行层整合计划

本页是后续实现的总蓝图：把旧的“判断器 + 故障动作”模式升级为“事件发现器 + 法典仲裁 + 事件执行层 + 上报器”。目标是让所有执行都可仲裁、可追踪、可恢复，同时保证软端 nodemanager 与硬件 safety 执行永远独立可靠。

1. 总体结论

本次设计的核心不是新增一个更大的“控制中心”，而是把职责切开：事件发现器只发现事实，事件仲裁器按法典统一认定并下发最终审判结果，事件执行层独立执行，最终由上报器统一对外表达。

设计红线：基础事件不能绕过法典直接执行；事件仲裁器不能接管具体执行细节；上报逻辑不能反向影响事件执行层。角色比喻是：事件仲裁器是法院，只下发最终判决；事件执行层是警察，负责执行判决。
事件发现器

只发布事实，不判断最终处置，不发执行命令。

事件仲裁器

法典规则匹配、组合升级、交叉仲裁、恢复降级。

事件执行层

合并软端 nodemanager 与硬件 safety 两条执行通道。

上报器

统一输出法典触发、执行结果、人工接管建议。

事件生命周期定义：事件发生即为错误开始，事件结束即为恢复结束。恢复不是另一条“反向命令”，而是事件从 active 变为 inactive 后，事件仲裁器基于剩余事实集重新下发最终判决。

2. 架构主链路

新链路以当前事实集为输入，以执行结果和上报载荷为输出。执行层与上报器并行消费仲裁结果，互不阻塞。

输入 / 事实 法典 / 仲裁 执行 上报 / 反馈
事件发现器 topic 频率、碰撞、小车状态、算法反馈、电池等模块只发现事实。
事件事实集 active_events 快照，按 event_key 去重并维护生命周期。
事件仲裁器 法典规则匹配、组合策略、交叉优先级、恢复状态机。
事件执行层 软端 nodemanager + 硬件 safety 独立执行。
上报器 触犯法典信息、执行结果、人工接管建议、上下文快照。 
EventDiscoverers -> EventFactStore -> EventArbiter -> Verdict / ExecutionPlan
                                           |                  |
                                           |                  +-> EventExecutor -> /safety_system/cmd
                                           |                                      -> /nodemanager/cmd
                                           |                                      -> ExecutionResult
                                           +---------------------> Reporter -> JSON topics

3. 命名与边界修正

这部分是后续代码、配置、文档需要统一的语言。命名先统一，后续实现才不会继续产生歧义。

旧说法新说法修正原因
判断器事件发现器它只能发现事实，不能直接认定最终错误或执行动作。
基础错误 / 基础故障事件事实事件 active 即错误开始，事件 inactive 即恢复结束；是否需要执行由法典决定。
组合故障仲裁事件仲裁器不只处理组合故障，也处理单事件、恢复、人工接管和最终审判结果。
硬件端事件执行层事件执行层执行层同时包含软端 nodemanager 和硬件 safety，两者统一归口。
安全执行上报 / 人工介入单独上报上报器上报统一包含法典触发信息、执行结果和人工接管建议。
控制源切换不归事件仲裁器直接接管。它属于 safety_emergency_executor 内部能力，事件仲裁器最多提供执行计划上下文。

4. 模块职责

发现与仲裁侧

topic_frequency_monitor

输出 topic 是否存在、是否有数据、频率是否低于阈值。

collision_monitor

输出碰撞区域、TTC、障碍事件是否 active。

vehicle_state_judge

输出小车运动状态事实，不直接申请人工介入。

algorithm_feedback_monitor

输出算法反馈事实，不携带执行命令。

event_arbiter

唯一的故障认定、法典命中、组合恢复和执行计划生成入口。

执行与上报侧

event_executor / nodemanager 通道

接收最终判决后发布 /nodemanager/cmd，负责节点重启、模块降级、模块恢复、执行 ACK。

event_executor / safety 通道

接收最终判决后发布 /safety_system/cmd，负责减速、缓停、急停、恢复、控制源内部策略、底盘输出。

reporter

统一上报法典触发、执行结果、人工接管建议和上下文证据。

外部云端/MQTT模块

订阅 reporter 输出，不直接依赖发现器或执行层内部状态。

5. 核心数据契约

建议先在内部 C++ 结构中落地，后续如需要跨进程隔离，再逐步消息化。

EventFact

事件发现器输出的事实状态；事件发生即错误开始，事件结束即恢复结束。

event_key: "collision.zone.emergency"
active: true
source: "collision_monitor"
severity_hint: "critical"
stamp: 1770000000.123
ttl_s: 0.5
evidence: { zone: "front", ttc_s: 0.35 }
CodexRule

法典规则，决定事实组合是否构成处置条件。

rule_id: "collision_emergency_stop"
priority: 100
when_all:
  - "collision.zone.emergency"
enter_hold_s: 0.0
clear_hold_s: 0.3
execution_level: "critical"
Verdict / ExecutionPlan

事件仲裁器输出的唯一最终审判结果，交由事件执行层执行。

plan_id: "plan-20260513-001"
rule_ids: ["collision_emergency_stop"]
targets: ["safety"]
safety_action: "EMERGENCY_STOP"
nodemanager_action: "NONE"
manual_takeover: true
ExecutionResult

事件执行层反馈的执行结果。

plan_id: "plan-20260513-001"
target: "safety"
accepted: true
executed: true
latched: true
reason: "emergency stop command sent"

6. 法典规则

法典是事件仲裁器的判断依据。单事件、组合事件、人工接管、软硬执行都应在法典中表达，而不是散落在各发现器里。

event_codex_rules:
  - rule_id: "collision_and_vehicle_abnormal"
    name: "碰撞风险叠加底盘反馈异常"
    priority: 120
    when_all:
      - event_key: "collision.zone.active"
      - event_key: "vehicle.motion.unexpected"
    enter_hold_s: 0.05
    clear_hold_s: 0.5
    min_hold_s: 1.0
    execution_plan:
      targets: ["safety", "nodemanager"]
      safety:
        action: "EMERGENCY_STOP"
      nodemanager:
        action: "RESTART_MODULE"
        module_keys: ["chassis_driver"]
    report_policy:
      human_takeover: true
      severity: "critical"
      reason: "collision risk with abnormal chassis feedback"
规则能力说明
when_all全部事件事实 active 才命中。
when_any至少一个事件事实 active 即命中。
min_match_count满足一定数量的事件集合，用于柔性组合策略。
enter_hold_s进入确认，过滤瞬时抖动。
clear_hold_s事件结束确认，确认通过即视为恢复结束，避免边缘反复跳变。
min_hold_s最小保持，保证执行层有足够时间完成动作。
execution_plan定义目标执行通道和意图，不直接描述底层执行细节。
report_policy定义上报严重等级、人工接管建议和展示原因。

7. 事件仲裁器

事件仲裁器每轮基于当前 active_events 快照重新计算，而不是只处理边沿事件。事件 active 表示错误开始，事件 inactive 表示恢复结束；仲裁器只根据当前仍然 active 的事件重新下发最终判决。

收集事实 过滤过期事实，按 event_key 建立 active set，并维护事件生命周期。
匹配法典 计算候选规则，维护 pending / active / clearing 状态。
输出计划 根据赢家规则生成唯一 Verdict / ExecutionPlan，只交给事件执行层。
场景仲裁行为
单事件出现也必须命中单事件法典规则后才生成执行计划。
多个事件逐步出现先按当前事实集执行，满足组合后升级为组合计划。
多个事件同批出现直接匹配组合规则，避免先输出一堆单项动作。
组合规则交叉按 priority -> execution_level -> matched_count -> active_prefer -> rule_id 稳定选赢家。
事件结束事件 inactive 即恢复结束；随后重新仲裁剩余 active 事实集，而不是简单发送旧错误恢复。

8. 事件执行层

事件执行层是执行边界，不是判断边界。它负责把执行计划转换为真实动作，并回传执行结果。软端和硬件通道统一归口，但内部必须保持独立。

软端通道：nodemanager

负责模块重启、节点管理、模块降级、模块恢复、节点执行结果 ACK。

/nodemanager/cmd cooldown multi-key merge
硬件通道：safety

负责减速、缓停、急停、恢复、控制源内部策略、最终底盘 /command 输出。

/safety_system/cmd latched stop control source internal
事件执行层必须保证幂等、锁存、超时、冷却时间和发布级执行反馈。当前 safety / nodemanager 下游尚未提供独立 ACK topic，因此现阶段上报的是发布结果、锁存状态和冷却抑制状态；后续只要下游补 ACK，执行层再接入真实确认即可。即使上报器异常，已经进入执行层的安全动作也不能被影响。

9. 上报器

人工接管上报统一并入上报器。发现器不直接申请人工介入，执行层不直接对外解释原因，上报器负责把法典触发和执行结果组织成外部可读消息。

上报内容来源说明
触犯法典信息事件仲裁器规则 ID、规则名称、触发事件集、严重等级、原因。
执行计划事件仲裁器计划 ID、目标通道、安全动作、节点动作、人工接管建议。
执行结果事件执行层发布结果、锁存状态、冷却抑制状态；真实 ACK 等下游接口提供后接入。
上下文快照事件事实集速度指令、底盘反馈、碰撞状态、topic 健康状态等证据。
 
{
  "event_type": "codex_rule_triggered",
  "rule_id": "collision_and_vehicle_abnormal",
  "severity": "critical",
  "manual_takeover": true,
  "execution": {
    "plan_id": "plan-20260513-001",
    "targets": ["safety", "nodemanager"],
    "results": [
      {"target": "safety", "executed": true, "latched": true},
      {"target": "nodemanager", "accepted": true, "executed": false}
    ]
  }
}

10. Topic 与配置

第一阶段需要明确执行链路方向：事件仲裁器不直接发布 /safety_system/cmd 或 /nodemanager/cmd。它只发布最终审判结果给事件执行层；事件执行层再负责应急措施执行和 nodemanager 命令发布。

接口阶段语义
/monitor/*_state保留并收敛事件发现器输出事实状态。
/nav2_monitor/status保留系统周期状态，后续应包含事件事实摘要和法典状态。
/nav2_monitor/fault_event语义更新从旧故障边沿逐步演进为法典触发事件。
/event_executor/verdict建议新增事件仲裁器输出的最终审判结果，事件执行层唯一输入。
/safety_system/cmd保留事件执行层硬件通道输出，不能由事件仲裁器直发。
/nodemanager/cmd保留事件执行层软端通道输出，不能由事件仲裁器直发。
/event_executor/plan兼容命名如果使用 plan 命名，也必须表达“最终判决”，语义等同 /event_executor/verdict。
/nav2_monitor/reporter/event_json保留并增强上报器输出法典触发 + 执行结果。
/nav2_monitor/reporter/human_takeover_json建议新增人工接管建议统一出口。
配置继续统一到 /opt/ry/config/Monitor/。仓库内 config/Monitor/ 作为运行配置镜像，用于版本管理和审查。

11. 实施阶段

阶段 1

文档与术语统一

更新 README、项目架构、接口说明、组合策略页，统一为事件发现器、事件仲裁器、事件执行层、上报器。

阶段 2

数据结构落地

新增或重构 EventFact、CodexRule、Verdict/ExecutionPlan、ExecutionResult 内部结构。

阶段 3

发现器改造

各判断器只输出事实 active / inactive。active 表示错误开始，inactive 表示恢复结束；移除直接执行、直接恢复和直接人工接管上报。

阶段 4

事件仲裁器实现

基于 active event 快照匹配法典，处理组合、交叉优先级、恢复状态机和唯一执行计划。

阶段 5

事件执行层整合

把 nodemanager 与 safety 归入事件执行层双通道，由事件执行层发布 /nodemanager/cmd 和 /safety_system/cmd，统一冷却、锁存和发布级执行结果；真实 ACK 等下游接口提供后接入。

阶段 6

上报器整合

人工接管、法典触发、执行结果、上下文快照全部通过上报器输出。

阶段 7

实机验证

用仿真事件和真实 topic 验证发现、仲裁、执行、反馈、上报完整闭环。

12. 测试验收

测试类型覆盖内容通过标准
单事件测试单个事件 active / inactive只有命中法典后才输出执行计划。
组合测试A+B、B+C、A+B+C组合规则优先级稳定，执行计划唯一。
逐步出现A 先出现，B 后出现先按 A 仲裁，满足 A+B 后升级组合计划。
同批出现A+B 同一周期出现直接进入组合规则，不输出多条单项执行。
恢复降级B 事件结束，A 仍 activeB 的恢复已随事件结束完成，系统重新基于剩余事实集仲裁，不乱发恢复命令。
执行失败safety 或 nodemanager 发布失败 / 后续 ACK 失败现阶段上报发布结果；下游 ACK 接口接入后，上报器包含执行失败原因，执行层状态不丢失。
高负载CPU 满载、topic 抖动事件执行层不被上报器或发现器阻塞。

13. 风险与约束

迁移期双语义风险

旧 FaultInfo 与新 EventFact 并存时，必须明确谁有执行权。迁移期也不能允许旧路径直接执行。

执行层被误耦合风险

上报器、仲裁器、发现器都不能在执行层内部加入阻塞逻辑，执行必须永远独立。

规则膨胀风险

法典规则会增长，需要规则命名、分组、优先级和审查流程，避免后期难维护。

恢复语义风险

事件结束即恢复结束。恢复不是反向执行旧动作，而是基于当前仍 active 的事实集重新计算执行计划。

最终验收标准：事件发现器只发现事实；所有执行均由法典仲裁产生；事件执行层软硬通道独立可靠；上报器统一表达触发原因、执行结果和人工接管建议。
